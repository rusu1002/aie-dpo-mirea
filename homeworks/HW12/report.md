# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- В данной работе решалась задача прогнозирования временного ряда с использованием корректной временной валидации. Использовался датасет S12-hw-dataset.csv, содержащий ежедневные наблюдения целевой переменной target (продажи). Прогнозировалось одно значение вперед (горизонт = 1 день).
- Разбиение данных выполнено строго по времени: 70% на обучение, 15% на валидацию и 15% на тест. Такой подход гарантирует отсутствие утечки информации из будущего в прошлое.

- Сравнивались четыре подхода:
    - B1 (Naive-last): прогноз = последнее известное значение
    - B2 (Moving-average): прогноз = скользящее среднее за 7 дней
    - B3 (Ridge-lag-features): линейная модель с регуляризацией на лаговых, rolling- и календарных признаках
    - R1 (GRU-forecast): рекуррентная нейросеть на оконном представлении ряда

- Лучший результат на валидации показала модель Ridge, что подтвердилось на тесте. GRU показала сопоставимые результаты, но уступила по стабильности.

Уровень шума умеренный, пропуски отсутствуют. Основная сложность прогноза — улавливание тренда и сезонности без использования внешних признаков.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: Temporal split
- Какие доли или интервалы использовались:
    - Train: первые 70% данных
    - Validation: следующие 15% данных
    - Test: последние 15% данных
- Почему `random split` здесь некорректен: Random split здесь некорректен, так как модель, обученная на перемешанных данных, могла бы "увидеть" будущие значения во время обучения, что дало бы нереалистично оптимистичные оценки качества на тесте. В реальном прогнозировании доступны только прошлые данные.

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались: 1, 2, 3, 7, 14, 28 дней
- Какие rolling-признаки использовались: среднее и стандартное отклонение за 7 и 28 дней (с shift(1), чтобы избежать утечки)
- Какие календарные признаки использовались: dayofweek (0-6), month (1-12)
- Как обрабатывались пропуски после построения лагов/rolling: после построения лагов и rolling-окон первые 28 строк содержат NaN и были удалены
- Использовалось ли масштабирование и где именно оно обучалось: для Ridge использован StandardScaler, обученный только на train; для GRU масштабирование также выполнялось только по train

## 5. Модели и эксперименты (B1, B2, B3, R1)

Опишите коротко и сопоставимо:

- B1 (`naive-last`): Прогноз = значение lag_1. Простейший baseline для проверки наличия временной структуры.
- B2 (`moving-average`): Прогноз = скользящее среднее за 7 дней (rolling_mean_7). Учитывает краткосрочную динамику.
- B3 (`ridge-lag-features`): Линейная модель с L2-регуляризацией (alpha=1.0). Использует все лаговые, rolling- и календарные признаки.
- R1 (`gru-forecast`): GRU с 2 слоями и 64 скрытыми нейронами.

Дополнительно:

- Основная метрика выбора лучшей модели: MAE
- Дополнительные метрики:
- `window_size` для `GRU`: 28
- Batch size: 64
- Максимальное число эпох: 30
- Optimizer: Adam
- Learning rate: 5e-4
- Как сохранялась лучшая модель: Лучшая модель сохранялась по минимальному val_loss

## 7. Анализ

(8-15 предложений)

Нужно прокомментировать:

- Наивный baseline (lag_1) дал большую ошибку — простое перенос последнего значения не учитывает ни тренд, ни сезонные паттерны.

- Moving average (7 дней) оказался даже хуже наивного baseline, что объясняется сглаживанием: на трендовом ряду среднее за неделю систематически отстает от фактических значений, особенно при восходящем тренде.

- Ridge на признаках показала значительное улучшение. Лаги и rolling-признаки позволили модели уловить недельную сезонность, а календарные признаки помогли зафиксировать разницу между буднями и выходными.

- GRU показала результат, сопоставимый с Ridge. Однако GRU не превзошла Ridge, что может объясняться:
    - Достаточностью линейной зависимости между лагами и целевой переменной
    - Относительно малым объемом данных (GRU лучше работает на больших выборках)
    - Наличием четкой сезонности, которую хорошо ловят явные dayofweek-признаки

- Тренд и сезонность хорошо видны в данных: восходящий тренд + недельные циклы. Ridge справился с этим благодаря лагу 7 и dayofweek. GRU смогла извлечь сезонность из самой последовательности значений.

- Потенциальные утечки данных: при построении rolling-признаков использовался shift(1), чтобы избежать включения текущего значения. Масштабирование всегда выполнялось только по train. Temporal split обеспечил отсутствие утечки по времени.

- Выбор по validation, а не по test принципиален, так как test должен оставаться "нетронутым" до финальной оценки. Если бы мы подбирали гиперпараметры по test, это привело бы к переобучению под конкретный тестовый период.

- Наиболее показательные ошибки возникают в моменты резких изменений тренда или выбросах.

## 8. Итоговый вывод

(3-7 предложений)

- В качестве базового подхода для данной задачи я бы выбрал Ridge на лаговых, rolling- и календарных признаках. Он показал наилучший результат, стабилен, легко интерпретируем и не требует сложной настройки.
- Простой baseline (наивный или moving average) достаточно, только если ряд стационарен и не имеет выраженной структуры.
- GRU имеет смысл применять, когда данных достаточно много (тысячи наблюдений), есть сложные нелинейные зависимости, ручное построение признаков затруднено, важна автоматическая извлечение сезонных паттернов.
