# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- База знаний: Использован датасет train_data.csv, содержащий 350 документов на русском языке по тематике финансов, банковского обслуживания, кредитования, налогов и прав клиентов. Эта база выбрана из-за её практической применимости для построения системы помощи (RAG) клиентам банка.
- Чанкинг документов: Реализована функция chunk_text, которая делит текст на чанки по количеству слов с перекрытием (overlap), чтобы сохранить смысловую связь между соседними фрагментами.
- Векторизация: Использована модель sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2, обеспечивающая dense эмбеддинги для русского языка. В случае её недоступности предусмотрен fallback на TF-IDF.
- Базовая проверка retrieval: На тестовом наборе из 8 запросов базовая конфигурация показала идеальные результаты (hit@3, recall@3, MRR@3 равны 1.0), что говорит о высокой различимости документов в выбранной выборке.
- После обновления базы: Добавление двух новых документов и двух новых запросов снизило средние метрики (hit@3 стал 0.8). После переиндексации метрики вернулись к 1.0, что демонстрирует критическую важность актуальности индекса.
- Mini-RAG: Простая RAG-система, формирующая контекст из найденных чанков и извлекающая из него наиболее релевантные предложения на основе TF-IDF схожести с запросом.

## 3. База знаний и подготовка документов

- Предметная область: Финансовые консультации, банковские услуги, налогообложение самозанятых, возврат средств, социальные выплаты.
- Источник документов: train_data.csv из курса "AI Finance Hack 2025". Ссылка на google диск: https://drive.google.com/file/d/1ywQpkiMf5IUeucYHRS811TCI5Isv4oCU/view?usp=sharing
- Число исходных документов: 350 (2 использовано)
- Число чанков после разбиения: 128
- Какой способ чанкинга использовался: Фиксированное количество слов (word-based chunking) с перекрытием.
- Основные параметры чанкинга: chunk_size=30 слов, overlap=10 слов.
- Комментарий (3-6 предложений): Выбранная база знаний хороша для тестирования RAG, так как документы тематически узкие (финансы) и содержат специфическую лексику. Это делает retrieval более простым, но в то же время реальные запросы могут быть многокомпонентными. Разбиение на чанки по словам — грубый, но эффективный для начала метод, хотя он может разрывать предложения и терять контекст на границах.

## 7. Анализ

- База знаний удобна для первого знакомства с RAG: документы компактны и тематически различны. Чанкинг по фиксированному числу слов сработал из-за малого размера документов, но на длинных текстах будет разрывать смысловые блоки.
- Оценка retrieval показала, что даже простая модель эмбеддингов отлично различает документы. Слишком маленькие чанки ухудшают ранжирование из-за потери контекста. Обновление базы знаний выявило "слепоту" системы к новым данным и важность переиндексации.
- Ошибки mini-RAG связаны с примитивным генератором, а не с retrieval: система не ответила на вопрос о кодах платежей (низкая релевантность фрагмента по TF-IDF) и выбрала отрицающий ответ о возврате средств, игнорируя другие чанки с условиями возврата.
- Главное ограничение — экстрактивная генерация, не позволяющая синтезировать ответ из нескольких фрагментов. Примитивный реранжинг часто выбирает не самые информативные предложения.

## 8. Итоговый вывод

- В качестве базового retrieval-конвейера для подобной задачи можно оставить связку "multilingual MiniLM + FAISS (IndexFlatIP) + чанкинг с разумным перекрытием (например, 30/10)", так как это обеспечивает высокую точность и скорость на умеренных объемах данных. Главный урок по retrieval: его качество критически зависит от репрезентативности эмбеддингов и согласованности индекса с актуальной базой знаний. Что касается связи retrieval и качества RAG: даже идеальный retrieval не гарантирует хороший ответ, если генеративный компонент не способен эффективно использовать найденный контекст. Система будет настолько сильна, насколько сильны её самые слабые компоненты — и в данном случае слабым звеном оказалась генерация.
