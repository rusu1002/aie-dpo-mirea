- **Название проекта:** `Разработка Финансового RAG-Ассистента`
- **Автор:** `Русу Ксения Олеговна`
- **Группа:** `БФБО-01-22`

- **Краткое описание (2-4 предложения):**  
  `Проект посвящён разработке RAG-системы для ответа на вопросы пользователей по финансовой документации. Система использует гибридный поиск, объединяющий плотный (Dense Retrieval на эмбеддингах) и разреженный (BM25) ретривинг, а также реранжирование найденных документов для повышения качества ответов. Используется датасет, предоставленный на одном из хакатонов, открытые модели эмбеддер, реранкер и LLM. Результат – REST API сервис, который принимает вопрос пользователя и возвращает ответ, сформированный на основе найденного контекста.`

## 2. Структура проекта

Проект организован в следующей структуре:

- `requirements.txt` – зависимости проекта (библиотеки Python, необходимые для запуска).
- `report.md` – отчёт по проекту (постановка задачи, данные, эксперименты, результаты).
- `self-checklist.md` – чеклист самопроверки проекта перед сдачей.
- `README.md` – основная информация по работе с проектом.
- `data/` – данные:
  - `train_data.csv` – датасет со статьями;
  - `questions.csv` – датасет с вопросами.
- `notebooks/` – экспериментальные ноутбуки:
  - `01_EDA.ipynb` – EDA;
  - `02_baselines.ipynb` – бейзлайны, прототипы моделей, исследование методов.
- `evaluation/` – тестирование RAG и оценка результатов:
  - `generate_answers.py` – запуск RAG для ответов на вопросы;
  - `run_deepeval.py` – оценка качества ответов системы.
  - `analyse_results.py` – сохранение результатов, статистики, графиков.
- `src/` – основной код проекта:
  - `data/` – модули для работы с данными:
    - `chunking.py` – разбиение на чанки;
    - `loader.py` – загрузка данных;
    - `preprocessing.py` – предобработка данных.
  - `generation/` – модуль генерации ответа:
    - `generator.py` – генерация ответа через api LLM / fallback из контекста;
    - `llm_manager.py` – выбор доступной модели LMM через API.
  - `reranker/` – модуль реранжирования контекстов:
    - `reranker.py` – реранжирование контекстов через api Reranker / fallback без реранжирования;
    - `reranker_manager.py` – выбор доступного Reranker через API.
  - `retriever/` – модуль ретривинга контекстов:
    - `bm25.py` – инициализация BM25 ретривера;
    - `fusion.py` – hybrid ретривинг контекстов;
    - `vectorstore.py` – загрузка эмбеддера, создание эмбеддингов.
  - `service/` – запуск сервиса:
    - `service.py` – API сервис.
    - `user_service.py` – user-friendly интерфейс-оболочка для сервиса.
  - `utils/`:
    - `logger.py` – логирование;
    - `metrics.py` – мониторинг метрик системы.
  - `manage_models.py` – проверка доступности моделей LLM и Reranker через API.
  - `RAG_pipeline.py` – код основного пайплайна системы RAG.
  - `train.py` – код обучения эмбеддера для контекстов.
- `configs/` – конфигурационные файлы:
  - `.env.example` – ключи API для доступа к сторонним сервисам.
  - `config.py` – необходимые константы.
- `tests/` – тесты (юнит-тесты, простые проверки).
- `artifacts/` – сохранённые модели, отчёты, результаты экспериментов.

## 3. Требования и установка

### 3.1. Требования

- Python `>= 3.10`.

### 3.2. Установка окружения

```bash
# Перейти в папку проекта
cd project

# Создать виртуальное окружение (опционально, но рекомендуется)
python -m venv .venv

# Активировать окружение:
# Windows:
.venv\Scripts\activate
# Linux / macOS:
source .venv/bin/activate

# Установить зависимости
pip install --upgrade pip
pip install -r requirements.txt
```

Для работы системы необходимо создать файл .env на основе шаблона:
```bash
cp .env.example .env
```
или на Windows:
```bash
copy .env.example .env
```
В файле необходимо указать:
```bash
LLM_API_KEY=<ваш ключ для доступа к LLM API>
RERANKER_API_KEY=<ваш ключ для доступа к reranker API>
```

## 4. Как запустить проект

### 4.1. Запуск индексации

```bash
cd project
python -m src.train
```

### 4.2. Проверка доступности сторонних сервисов

```bash
cd project
python -m src.manage_models
```

### 4.2. Запуск сервиса

```bash
cd project
uvicorn src.service.service:app --reload
```
Запуск user-friendly интерфейса (опционально; **после запуска сервиса**):

```bash
cd project
streamlit run src/service/user_service.py
```

- После запуска сервис будет доступен по адресу: http://127.0.0.1:8000;
- Эндпоинты:
  - GET /health: Проверка работоспособности сервиса.
  - POST /predict: Основной эндпоинт для получения ответа RAG.
  - GET /metrics: Метрики работы системы в формате Prometheus.
  - GET /docs: Автоматическая Swagger-документация FastAPI.
- как протестировать работоспособность: через эндпоинт /health:
```bash
curl http://localhost:8000/health
```

## 8. Ограничения и дальнейшая работа

- В рамках проекта была реализована базовая RAG-система для ответа на финансовые вопросы пользователей, однако некоторые возможности не вошли в текущую версию;
- Основные ограничения:
  - сервис разворачивается локально и не контейнеризирован с помощью Docker;
  - не реализовано хранение истории запросов и ответов;
  - генерация выполняется через внешний LLM API, поэтому качество ответов зависит от доступности и характеристик выбранной модели;
  - эксперименты проведены преимущественно для ретривинг части, без детального сравнения различных генеративных моделей;

- Проект может быть расширен в нескольких направлениях:
  - добавить Docker-контейнеризацию;
  - организовать хранение истории запросов и ответов в базе данных;
  - провести сравнительный анализ различных LLM-моделей для генерации ответов;
  - протестировать несколько реранкеров и оценить их влияние на качество поиска;
  - добавить автоматическую оценку качества ответов в пайплайн с использованием Retrieval Metrics и LLM-as-a-Judge;
  - реализовать автоматическое обновление индексов;
  - добавить веб-интерфейс для взаимодействия с системой;
  - расширить набор экспериментов и провести более детальное исследование влияния параметров чанкинга, эмбеддеров и методов гибридного поиска на качество ответов.

`Комментарий автора: В процессе участия в хакатоне мне не удалось создать работоспособную и качественную систему, из-за недостатка знаний, времени и сил, и задача отложилась на неопределенное время. Поэтому я решила выбрать ее в качестве темы для выпускного проекта, чтобы "закрыть гештальт". На момент начала работы над проектом у меня был реализован только препроцессинг данных и поиск с помощью BM25, что вошло в работу как один из baseline'ов.`
